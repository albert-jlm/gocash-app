## Decision Records

### DR-001 — Next.js 15 with Static Export (not SSR)

**Decision:** Use Next.js 15 App Router with `output: 'export'` (static HTML/CSS/JS only).

**Why:**
- Capacitor requires a directory of static files to bundle into the native app. SSR requires a running Node.js server — Capacitor cannot bundle a server.
- Static export means the app can also be hosted on any CDN (Cloudflare Pages, Netlify, GitHub Pages) with zero server cost.
- All backend logic lives in Supabase Edge Functions, so there's no need for Next.js API routes.

**Trade-offs accepted:**
- No `next/image` optimization (use `<img>` or a CDN transform instead)
- No server components with data fetching (all data fetching is client-side via Supabase SDK)
- No incremental static regeneration (ISR)
- Middleware runs at build time only

**Alternatives rejected:**
- **Expo/React Native** — separate UI codebase, high duplication, rejected
- **Flutter** — separate codebase, not web-first, rejected
- **TWA (Trusted Web Activity)** — Play Store only, no iOS App Store, rejected
- **Next.js with SSR on Vercel** — requires server hosting, incompatible with Capacitor static export constraint

### DR-002 — Capacitor 6 (not React Native / Expo)

**Decision:** Wrap the Next.js static export in a Capacitor native shell.

**Why:**
- One UI codebase ships to 3 platforms: iOS App Store, Google Play Store, and web PWA.
- Capacitor bridges native APIs (camera, push notifications, haptics) to the same JavaScript running in the WebView — no conditional code needed.
- `@capacitor/camera` handles photo picker on iOS, camera intent on Android, and WebRTC on web automatically.
- Deep integration with existing web tooling (npm, TypeScript, Tailwind).

**Key configuration:**
```typescript
// capacitor.config.ts
{
  appId: 'com.gocash.tracker',
  webDir: 'out',              // Next.js static export output
  server: { androidScheme: 'https' }  // Required for Supabase auth cookies
}
```

**Risk:** WebView rendering differences between iOS Safari and Android Chrome. Mitigation: test on both platforms early.

### DR-003 — Supabase (Self-Hosted) as Full Backend

**Decision:** Use the self-hosted Supabase instance at `supabase.zether.net` for Auth, PostgreSQL, Storage, Realtime, and Edge Functions.

**Why:**
- Single backend covers auth, data, real-time, file storage, and server-side AI processing — no additional services needed.
- Supabase RLS handles multi-tenant data isolation declaratively — no application-level tenant filtering needed.
- Supabase Realtime uses PostgreSQL CDC (Change Data Capture) for live wallet and transaction status updates.
- Edge Functions (Deno) can call OpenAI directly, keeping API keys server-side.

**Multi-app strategy:** One instance, multiple PostgreSQL schemas. GoCash uses the `gocash` schema. The Supabase client is scoped: `createClient(url, key, { db: { schema: 'gocash' } })`.

**Risks:**

| Risk | Severity | Mitigation |
|---|---|---|
| Homelab server goes down (single point of failure) | Medium | Acceptable for v1 (single operator). Add Supabase Cloud fallback in v2. |
| Server RAM pressure (2.7Gi swap in use) | Medium | Supabase is already running — no additional RAM cost for this app. |
| `auth.users` shared across all apps on the instance | Low | Acceptable — scoped by schema + RLS. Separate instance only if auth isolation required. |
| Self-hosted upgrades require manual maintenance | Low | Follow Supabase release notes; use `docker compose pull && docker compose up -d`. |

### DR-004 — AI Pipeline in Supabase Edge Functions (not n8n)

**Decision:** Replace the existing n8n workflow with two Supabase Edge Functions: `process-transaction` (Phase 1) and `confirm-transaction` (Phase 2).

**Why:**
- Eliminates homelab dependency on a running n8n instance — the app is fully self-contained.
- Edge Functions run on Deno (V8 isolates, ~50ms cold start) — fast enough for the processing pipeline.
- OpenAI SDK is fully compatible with Deno. API key lives in Supabase Edge Function secrets, never exposed to the client.
- Business logic in `lib/transaction-processing.ts` is portable TypeScript — unit-testable with Vitest, reusable across functions.

**OpenAI model choices:**

| Model | Use | Why |
|---|---|---|
| `gpt-4o` | OCR (vision) — extract text from screenshot | Highest accuracy on image inputs |
| `gpt-4.1-mini` | Date + reference number extraction from OCR text | Cheaper for structured text extraction; no image needed |

**Cost estimate per transaction:** ~$0.003–0.008 USD (GPT-4o vision) + ~$0.0002 (gpt-4.1-mini) ≈ **< $0.01 per transaction**.

**Alternatives rejected:**
- **n8n** — external dependency, homelab only, not distributable, removed
- **Next.js API routes** — requires SSR hosting, incompatible with static Capacitor export
- **AWS Lambda / Vercel Functions** — adds infrastructure complexity; Supabase Edge Functions are already available

### DR-005 — PostgreSQL Schema Isolation (not separate databases)

**Decision:** Each app on the shared Supabase instance gets its own PostgreSQL schema (`gocash`, future apps get their own).

**Why:**
- True namespace isolation — no table name conflicts between apps.
- `auth.uid()` works across all schemas (shared auth system).
- PostgREST can expose multiple schemas simultaneously via `PGRST_DB_SCHEMAS`.
- Zero additional RAM/CPU cost compared to spinning up a new Docker stack per project.

**Adding a new app:**
```sql
CREATE SCHEMA new_app;
GRANT USAGE ON SCHEMA new_app TO anon, authenticated, service_role;
-- + add 'new_app' to PGRST_DB_SCHEMAS in ~/homelab/supabase/.env
-- + docker compose restart rest
```

### DR-006 — TanStack Query + Zustand (not Redux / SWR / Context)

**Decision:** TanStack Query for server state (transactions, wallets), Zustand for client state (UI state, offline queue).

**Why:**
- TanStack Query handles caching, background refetching, and optimistic updates for Supabase data out of the box. It pairs naturally with `@supabase/supabase-js` queries.
- Zustand is minimal (< 1kB), React-free compatible, and works cleanly in a Capacitor WebView.
- No Redux boilerplate for what is ultimately a small mobile app.

**Pattern:**
```typescript
// Server state → TanStack Query + Supabase
const { data: wallets } = useQuery({ queryKey: ['wallets'], queryFn: () =>
  supabase.from('wallets').select('*')
})

// Client/UI state → Zustand
const { pendingUpload, setPendingUpload } = useAppStore()
```

### DR-007 — pnpm (not npm / yarn)

**Decision:** pnpm as package manager.

**Why:**
- Faster installs, content-addressable store (deduplication across projects).
- Strict dependency resolution catches phantom dependencies that npm silently allows.
- Compatible with all Next.js, Capacitor, and Supabase tooling.

## Constraints Locked In

These are **non-negotiable** — changing them requires a new DR:

1. **`output: 'export'` on Next.js** — required for Capacitor. No SSR. No API routes.
2. **OpenAI called from Edge Functions only** — API key never in client bundle.
3. **Wallet balances update in Phase 2 only** — after explicit operator confirmation. Never auto-committed.
4. **RLS on every Supabase table** — no table without a policy.
5. **One schema per app** on the shared Supabase instance.
